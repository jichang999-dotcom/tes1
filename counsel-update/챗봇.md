# [SOP] 구글챗 봇 기반 AI 리포트 고도화 자동화

> 본 문서는 `sop_AI리포트2_자동화.md` 시스템 위에 얹는 **n8n + 구글챗 봇 고도화 레이어**를 정의한다.
> 두 개의 신규 기능을 통합 관리:
> - **Part A**: 상담사 조치완료 자동화 (리셋 포인트 방식)
> - **Part B**: 키워드 사전 진화 파이프라인 (승인 없는 즉시 반영)

---

## 1. 문서 개요

- **목적**: 상담사·센터장이 구글챗 봇과 대화하여 리포트 품질을 실시간으로 개선하는 양방향 시스템 구축
- **공통 원칙**:
  - n8n이 **이벤트 허브** 역할 (구글챗 입력 수신 → 검증 → 시트 기록)
  - 로컬 배치(19:00)가 시트를 읽어 **다음 리포트에 반영**
  - 승인 단계 최소화 → 자동 검증 + 롤백으로 안전성 확보
- **연계 시스템**: `sop_AI리포트2_자동화.md` 6단계(`build_html.js`) + 2단계(`analyze_sentiment.js`) 확장
- **실행 주기**: 실시간 이벤트 수신 + 기존 일일 배치 활용

|---|---|---|

## 5. Part A — 상담사 조치완료 자동화

### 5-1. 핵심 컨셉

**"조치완료 = 리셋 포인트"**

```
학생 S12345 타임라인:
  3/20  상담: "환불 고민" (Strong)     ┐
  3/25  상담: "수업 불만" (Weak)       ├─ 과거 (리포트에서 숨김)
  4/01  상담: "퇴소 의사" (Strong)     ┘

  4/02  상담사 "S12345 조치완료" ★ 리셋 포인트

  4/05  상담: "다시 환불" (Strong)     ┐ 새로 누적 (재발 경보)
  4/10  상담: "이탈 재언급" (Strong)   ┘
```

### 5-2. n8n 워크플로우 (Part A)

```
[1] Google Chat Trigger
      │
      ▼
[2] Function: 파서
      ├─ 정규식: /^(S\d+)\s+(조치완료|조치취소)(?:\s+(.+))?$/
      └─ 미매칭 → 사용법 안내
      │
      ▼
[3] Switch: 액션 분기
      ├─ 조치완료 → 등록 플로우
      └─ 조치취소 → 취소 플로우
      │
      ▼
[4-등록] 검증 (학생코드 형식, 중복 active 체크)
      │
      ▼
[5-등록] Sheets append (상태=active)
      │
      ▼
[6-등록] DM: "✅ 접수 (다음 배치부터 반영)"

────────────────────────────────────────

[4-취소] 해당 학생 active 행 검색
      │
      ▼
[5-취소] 권한 확인 (원 등록자 or 관리자)
      │
      ▼
[6-취소] Sheets update (cancelled + 취소일시 + 사유)
      │
      ▼
[7-취소] DM: "✅ 조치취소 처리됨"
```

```javascript
const { reset_map } = JSON.parse(fs.readFileSync('data/action_log.json'));

function filterByReset(records, std_cd) {
  const reset = reset_map[std_cd];
  if (!reset) return records;
  return records.filter(r => r.cns_dt > reset.reset_at);
}

const effectiveRecords = filterByReset(studentRecords, std_cd);

if (reset_map[std_cd] && effectiveRecords.length === 0) {
  renderDoneCard(std_cd, reset_map[std_cd]);  // 조치완료 아카이브
} else if (reset_map[std_cd] && effectiveRecords.length > 0) {
  renderCard(effectiveRecords, { warning: '조치 후 재발생' });
}
```

### 6-4. 머지 로직

#### `fetch_keyword_override.js` (Step 1.5)
```javascript
copyFileSync('keywords.json', `keywords.json.backup_${today}`);
const overrides = await readSheet('키워드 오버라이드', { status: 'applied' });

for (const req of overrides) {
  const { 센터코드, 분류, 카테고리, 키워드, 액션 } = req;
  const target = keywords[분류].by_center[센터코드] ||= { add: {}, exclude: {} };
  
  if (액션 === '추가') {
    target.add[카테고리] ||= [];
    if (!target.add[카테고리].includes(키워드)) target.add[카테고리].push(키워드);
  } else if (액션 === '제외') {
    target.exclude[카테고리] ||= [];
    if (!target.exclude[카테고리].includes(키워드)) target.exclude[카테고리].push(키워드);
  } else if (액션 === '복원') {
    if (target.exclude[카테고리]) {
      target.exclude[카테고리] = target.exclude[카테고리].filter(k => k !== 키워드);
    }
  }
}
writeFileSync('keywords.json', JSON.stringify(keywords, null, 2));
```

#### `analyze_sentiment.js` 수정
```javascript
function resolveKeywords(keywords, cmp_cd, classification) {
  const global = keywords[classification].global || {};
  const override = keywords[classification].by_center?.[cmp_cd] || {};
  const adds = override.add || {};
  const excludes = override.exclude || {};
  
  const merged = {};
  const allCats = new Set([...Object.keys(global), ...Object.keys(adds)]);
  for (const cat of allCats) {
    const base = [...(global[cat] || []), ...(adds[cat] || [])];
    const excludeSet = new Set(excludes[cat] || []);
    merged[cat] = base.filter(k => !excludeSet.has(k));
  }
  return merged;
}
```

### 6-5. 롤백 메커니즘

```
"키워드 롤백 REQ-2026-0415-001"
  └─▶ 원 요청 조회
        └─▶ 역연산 요청을 새 행으로 append
              ├─ 원본 "추가" → add에서 제거
              └─ 원본 "제외" → exclude에서 제거 (= 복원)
        └─▶ 원 요청 상태 → rolled_back
```

권한: 원 요청자 + 관리자만.

### 6-6. 엣지 케이스 (Part B)

| 케이스 | 처리 |
|---|---|
| 전사 global 수정 요청 | 완전 차단 + "PR 안내" |
| global에 없는 키워드 제외 | "이미 미활성" 안내 |
| 이미 제외된 키워드 재제외 | "이미 제외됨" 회신 |
| add에 있는 키워드 제외 | "add에서 제거 권장" 안내 |
| exclude + add 동일 키워드 충돌 | exclude 우선 (제외 이김) |
| 존재하지 않는 센터 | 거절 |
| 카테고리 오타 | 유사 카테고리 3개 제안 |

## 7. 자동 검증 규칙 (공통 안전장치)

### Part A 검증

|---|---|
| 학생코드 형식 | `S\d+` 또는 실제 std_cd 패턴 |
| 권한 | 등록: 누구나 / 취소: 원 등록자 + 관리자 |
| 중복 방지 | 동일 학생 active 존재 시 확인 |

### Part B 검증
| 항목 | 기준 | 조치 |
|---|---|---|
| 길이 | 2~20자 | 미달/초과 거절 |

| 권한 | 요청자의 센터 권한 | 타 센터 차단 |
| 전사 global | 봇 수정 시도 | 완전 차단 |
| 제외/복원 사유 | 누락 | 재요청 |
| 카테고리 존재 | keywords.json | 유사 카테고리 제안 |
