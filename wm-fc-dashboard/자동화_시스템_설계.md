## 3. 전체 플로우 (사용자 제공 8단계)

```
[Win 스케줄러 월~금 08:30 KST]
   │
   ▼
① schedule-gate        평일·공휴일·중복실행·자격증명 게이트
   │
   ▼
② db-extractor         v4 쿼리 실행 → 원본 + CSV 저장
   │
   ▼
③ data-archiver        runId 폴더 동결, SHA-256 매니페스트
   │
   ▼
④ dashboard-builder    템플릿 + 데이터 정적 주입 → build/_auto_stage.html ★ (파이프라인 전용 staging)
   │
   ▼
⑤ integrity-verifier   결정적(L1/L2/L3) + AI 시각(L4) + 종합(L5)

   │
   ▼
⑥ publish-gateway      pass → _auto_stage.html → wm_fc_desh.html 승격(copy) → github-publisher
                       fail → dashboard-repairer (AI, 1회) → ⑤ 재검증
                              그래도 fail → 롤백 + 단테 Slack 알림 (발행 보류, 직전판 유지)
   │
   ▼
⑦ propagation-waiter   60초 대기 (GitHub Pages CDN 전파)
   │
   ▼
⑦b verification-report 검증 리포트 PDF 생성 + GitHub 발행 (실패해도 파이프라인 무영향)
   │
   ▼
⑧ notifier             발행 성공=#wm_fc_dash 1건 / 판단 필요=단테 자동화 채널 1건
```

> **2-stage publish gate (2026-05-29 확정, 2026-06-08 staging 파일 분리):** ④는 항상 파이프라인 전용 staging `build/_auto_stage.html`에 먼저 빌드한다. `build/wm_fc_desh.html`(운영 발행 파일)은 ⑤ 전계층 통과 후 ⑥에서만 갱신된다. 따라서 운영 파일은 항상 "마지막으로 검증 통과한 빌드"의 상태로 보존된다. ⚠ `test_dash.html`은 사용자 개발 샌드박스로 파이프라인이 절대 건드리지 않는다(2026-06-08 데이터/UI 분리, 상단 ★규약 참조).

## 4. DB 연결 & 추출

### 4-1. 자격증명 출처

- 본 시스템은 동일 `.env`를 읽는다. 별도 자격증명 분리 불필요.

### 4-2. 쿼리

- 단일 파일 `wm_fc/config/sql/v4_kpi.sql`에 보관 (v4 쿼리 원문 그대로)
- 기간: `class_ogn_ym BETWEEN '202301' AND DATE_FORMAT(NOW(), '%Y%m')` (현재 달력월까지 자동 갱신, 2026-05-28 동적 처리로 변경)
- 출고 기간: `mtr_out_dt BETWEEN '2023-01-01 00:00:00' AND LAST_DAY(NOW()) 23:59:59` (현재 달력월 말일까지)
- 화이트리스트: `cmp_cd IN ( ... 102개 ... )` (쿼리 내장)
- 결과 구조: **단일 UNION 결과**. pk별로 과목수 행(class_cn>0, mtr_cn=0) + 출고수 행(class_cn=0, mtr_cn>0)이 별도 행으로 떨어진다. 대시보드는 필터 시점에 합산한다.

## 5. 데이터 보관 (스냅샷)

- `runId = YYYYMMDD-HHMM` (KST). 폴더 1개당 한 run.
- 각 run 폴더는 **immutable** — 한번 쓰면 수정 금지. `manifest.sha256`에 모든 파일 해시 기록.
- `wm_fc/data/current/` 는 **마지막 검증·발행 통과판**의 미러. ⑥ 통과 시에만 갱신.
- 보존 정책: **무제한 보관** (모든 run 계속 보존, 자동 삭제 없음 — 2026-06-01 사용자 결정). `03_data_archiver.js`의 `RETENTION_DAYS=0`. 0보다 큰 값으로 바꾸면 그 일수 경과분 자동 삭제로 전환.
- 의미: "그날 대시보드가 왜 그 숫자였는지"를 사후에 재현 가능 → 검증·보안의 흔적(audit trail).

### 6-2. 기술 스택

### 6-3. 데이터 주입 방식 (수동 업로드 → 자동 주입)

샘플은 사용자가 브라우저에서 `data.csv`를 업로드하는 구조다. 자동화에서는 이를 **빌드 시점 정적 주입**으로 바꾼다.

```html
<!-- dashboard/template.html 빌드 결과 -->
<script id="__DATA__" type="application/json">[ ... rows ... ]</script>
<script>const RAW = JSON.parse(document.getElementById('__DATA__').textContent);</script>
```

- 업로드 버튼은 **유지**(원장이 수동 점검 시 다른 CSV로 교체 가능). 단, 기본값은 정적 주입 데이터.
- 빌드 시점 메타: 생성 ts, runId, 행수, 쿼리 해시를 헤더 우측에 작게 표시 (디버깅·신뢰성용)

| **종합 대시보드** | KPI 카드 4종(총 과목수·총 출고수·평균 출고율·MoM 증감률), 연·월별 추이 라인 차트(과목수 vs 출고수), 협의회 비교, 프로그램 순위 |
| **이상 징후 대응 관리** | 출고율 미달(`< minRateThreshold`, 기본 90%) / 초과(`> maxRateThreshold`, 기본 110%) 센터 목록, 과목 급감(`MoM ≤ -15%`) 위험 센터, 임계치 슬라이더, 조치 체크박스 |
| **센터별 상세 현황** | 센터 카드(아코디언), 정규 vs 특강 분할, 프로그램별 명세, 센터 월별 추이 |
| **과목 및 프로그램 분석** | 수학 vs 과학 점유율, 정규 vs 특강 점유율, 프로그램별 출고율 분포 |

### 6-5. 필터 (8개)

### 6-6. 임계치 (사용자 조정 슬라이더)

이상 징후 탭의 슬라이더 2개로 운영자가 직접 조정한다.

| 슬라이더 | 조정 범위 | 기본값 |
|----------|----------|--------|
| 미달 경고선 | **50% ~ 95%** | **90%** |
| 초과 경고선 | **105% ~ 150%** | **110%** |
| 정상 구간 | 미달선 ~ 초과선 | (계산값) |

추가 임계치(슬라이더 외):
- `dropRiskThreshold = -15` (과목 급감 MoM %)

저장 방식:
- 기본값·범위 정의: `wm_fc/config/thresholds.json` (**Phase 1부터 외부화** — 사용자가 슬라이더로 조정해야 하므로 미루지 않음)
- 사용자 조정값: 브라우저 `localStorage` (새로고침해도 유지)

알림 분류 로직:
- 출고율 < 미달선 → 🔴 미달
- 출고율 > 초과선 → 🟠 초과
- 그 외 → 🟢 정상
- 분모(과목수)=0 인데 분자(출고수)>0 → 🟠 초과(과잉 출고)로 강제 분류 (단순 0% 표시 X). 화면 표시는 `"—"` + ⚠️ 뱃지

## 7. ★ 검증·보안 게이트 (L1~L5)

플로우 ⑤번. 모든 검증은 **`build/_auto_stage.html`(파이프라인 전용 staging)** 에서 수행한다. 운영 파일 `build/wm_fc_desh.html`은 L1~L4 모두 pass + L5 종합 통과 후 ⑥에서만 승격된다. (`test_dash.html`은 사용자 샌드박스라 검증 대상이 아니다 — 2026-06-08 분리)

| 계층 | 검증 내용 | 실패 처리 |
|------|-----------|-----------|
| **L1 추출 정합성** | 0건 방어, NULL·음수 방어, 센터 화이트리스트(102개), 음의 `class_cn/mtr_cn` 검출, 출고율 ∞(class_cn=0인데 mtr_cn>0) 탐지 | 중단 |

### 7-2. L4 AI 시각·상호작용 검증 명세 (2026-05-29 사용자 확정)

**실행 형태:** Claude Code 헤드리스(`claude -p`) + Playwright. AI는 단순 selector 매칭이 아닌 사용자처럼 행동하여 화면이 합리적으로 갱신되는지 점검한다. `--max-turns 60`(2026-06-15 40→60 상향, 턴 소진 실패 방지) · 타임아웃 20분.

### 7-2-1. L4 오탐 방지 & 자가 진화 (2026-06-15 신설)

**배경:** L4 검사 AI는 매 회차 Playwright 검증 스크립트를 새로 작성한다. 그 스크립트 자체의 버그(예: 음수 칩 `▼` 파싱 누락, `fullReset` 시 활성 탭 미복원)로 **멀쩡한 항목이 fail**로 찍히면, AI가 디버깅·재작성을 반복하다 `--max-turns`를 소진해 보고서 미생성 → 발행 차단 → 알림 누락이 발생했다(2026-06-15 사고). 데이터/대시보드 문제가 아니라 **검증기 측 오탐**이 근본 원인.

**(1) 오탐 방지 함정노트** — `scripts/l4_checklist.md` 의 "⚠️ 오탐 방지 — 알려진 함정" 섹션. AI가 스크립트 작성 전 정독해 같은 함정을 피한다. 현재 5건(2026-06-15 음수 칩 파싱·fullReset 탭 복원 2건 보강).

**(2) 자가 진화 (사후검토 모델, 2026-06-15 사용자 확정)** — 새 함정을 사람 개입 최소로 누적·반영:

```
L4 보고서 new_traps[]  →  trap_ledger.json 누적(key별)  →  2회 이상 재현 시
   (검사 AI가 자가 보고)        (lib/l4_reflect.js)         l4_checklist.md 자동 append
                                                                      │
   담당자 확인·수동 롤백  ◀──  ⑧ 담당자 채널 알림(변경 시에만)  ◀──────┘
```

- **트리거:** `l4_runner.js`가 L4 보고서 파싱 성공 후 `reflectTraps(runId, report)` 호출. 실패해도 try/catch로 L4 판정엔 무영향.
- **AI 입력:** 보고서 스키마에 `new_traps[{key, item_ids, guide_text, evidence}]` 추가. 함정노트에 **없던** 오탐만 보고(진짜 결함은 fail로 남김).
- **안전장치 (4종):**
  1. **추가-only** — 함정노트에 append만. 기존 항목 수정·삭제는 **사람만** (시스템 자가변조 차단)
  2. **2회 재현 게이트** — 같은 `key`가 서로 다른 run에서 2회 이상 나와야 반영 (1회성 우연 배제)
  3. **롤백 스냅샷** — 변경 직전 원본을 `scripts/_backups/l4_checklist.<runId>.md`에 보관 + `data/guide_change_log.json` 감사추적 (git 미사용 환경)
  4. **변경 시에만 발송** — 함정이 추가된 구동에서만 ⑧이 담당자 채널(`SLACK_WEBHOOK_URL`)로 알림. 변경 없는 구동은 침묵(알림 피로 방지)
- **삽입 위치:** `l4_checklist.md` 의 `<!-- AUTO_TRAP_INSERT -->` 마커 위. 마커 삭제 금지.
- **산출물:** `data/trap_ledger.json`(후보 원장), `data/guide_change_log.json`(반영 이력), `data/runs/<runId>/guide_change.json`(이번 회차 변경 요약 — ⑧이 읽음).

> 위험 통제: 사후검토라 잘못 추가된 함정이 다음 구동까지 살 수 있으나, 노출 창은 최대 1회차(하루)로 제한되고, 함정노트는 "오탐 통과" 방향이라 최악도 L4 시각검사 한정(숫자 정합성은 결정적 L1~L3가 보증). 데이터 사고로 번지지 않음.

  - **B. 결정적 스텝 재실행 허용**: 상류 산출물(raw.jsonl / extract_meta.json / data.csv / manifest.sha256)이 손상으로 의심되면 직접 편집 금지, 대신 `node src/steps/02_db_extractor.js --runId <runId>` 또는 `node src/steps/03_data_archiver.js --runId <runId>`를 재실행하여 결정적으로 재생성. 결과는 DB에서 새로 가져온 값이라 위조 위험 없음
  - **금지**: `raw.jsonl` / `data.csv` / `extract_meta.json` / `config/sql/*.sql` 직접 편집은 절대 금지 (위조 차단)
- 안전장치:
  1. 수정 후 **⑤ 재검증 강제** (재실행하지 않으면 ⑥-A 진입 차단)
  2. 변경 diff → `repair_diff.json` (`code: {file→bytes}`, `upstream: {file→{regenerated, before_sha256, after_sha256}}`) 영구 보관
  3. **최대 1회 시도** (루프 방지) · Claude `--max-turns 40` · 타임아웃 10분 (②/③ 재실행 고려) ※2026-06-15 25→40 상향(턴 소진 실패 방지)
  4. 재검증도 fail → **코드만 자동 롤백** (`pre_repair_snapshot.json`에서 복원, 상류 재생성본은 DB 결정적이라 유지) → `current/` 직전판 유지 → 단테 자동화 Slack 채널 알림 (발행 보류)
  5. 코드도 상류도 변경 없음(`no_changes`) → AI가 자가복구 불가 판단 → 단테 채널 알림으로 사용자 판단 위임
  6. 모든 시도는 `repair_diff.json` / `repair_outcome.json` / `repair_claude_stdout.txt`로 영구 보관 (사후 검토)

### 8-2. `github-publisher`

- 자체 구현 (`wm_fc/src/lib/github.js`) — native fetch + Contents API (octokit 의존성 X)
- 자격증명: `.env`의 `GITHUB_TOKEN / GITHUB_OWNER / GITHUB_REPO / GITHUB_BRANCH(=main)`
- **발행 경로 (2026-05-29 갱신):** `jichang999-dotcom/tes1` 저장소의 `wm-fc-dashboard/wm_fc_desh.html`
- **공개 URL:** https://jichang999-dotcom.github.io/tes1/wm-fc-dashboard/wm_fc_desh.html
- 발행 흐름:
  1. **L1~L4 모두 pass인지 확인** (verification_report.json + l4_visual_report.json)
  2. **`build/_auto_stage.html` → `build/wm_fc_desh.html` 승격** (`fs.copyFile`)
  3. 기존 파일 SHA 조회 → PUT (sha 포함 시 update) → 응답에서 commit SHA 추출
  4. `current/wm_fc_desh.html` 갱신 → `publish_meta.json` 저장
- **409(SHA 낙관적 잠금 충돌) 재시도 (2026-06-02 추가, `putFile`):** 대용량(~19MB) 파일은 Contents API GET이 CDN 캐시로 **낡은 blob SHA**를 반환해 후속 PUT이 409로 거부되는 일이 있다. 이때 GitHub 409 본문(`"is at <실제SHA>"`)에서 진짜 현재 SHA를 파싱해 그 값으로 재PUT하며, 패턴이 없으면 캐시 무력화 재조회. 최대 4회 백오프 재시도(400ms×시도). 409 외 상태(401/403/422/5xx)는 즉시 실패. 반환 메타에 `attempts` 포함.
- 운영 파일 보존 원칙: 승격은 L4 통과 후에만 수행. 어떤 단계에서든 실패 시 `wm_fc_desh.html`은 **이전 발행 시점의 콘텐츠** 그대로 유지된다 (직전 발행물 보존)

### 8-3. `publish_design_doc` (설계 문서 발행, 2026-06-25 신설)

대시보드뿐 아니라 **본 설계서(`자동화_시스템_설계.md`)도 같은 GitHub 디렉토리에 공개**한다. 문서가 수정될 때마다 항상 최신본이 발행되도록 단일 진입점을 둔다.

- 구현: `src/publish_design_doc.js` — `src/lib/github.js`의 `getFile`/`putFile` 재사용 (octokit 의존성 X, 409 재시도 로직 그대로 활용)
- 대상 매핑(`DOCS` 배열): `wm_fc/자동화_시스템_설계.md` → `wm-fc-dashboard/자동화_시스템_설계.md`. 추후 문서 추가 시 이 배열에 한 줄만 추가
- **idempotent:** 로컬 파일의 git blob sha(`sha1("blob <len>\0" + content)`)를 원격 `contents.sha`와 대조해 **동일하면 skip**(불필요한 커밋 방지), 다를 때만 PUT. `--force`로 강제 재발행
- 공개 URL(마크다운 렌더): https://github.com/jichang999-dotcom/tes1/blob/main/wm-fc-dashboard/자동화_시스템_설계.md
- 산출물: `data/current/design_doc_publish_meta.json` (발행 시각·commit·skip 여부·URL)
- **발행 경로 (2종):**
  1. **수동 단발:** `node src/publish_design_doc.js` (더블클릭 `scripts/publish_design_doc.bat`) — 설계서 편집 직후 실행
  2. **동기화 연동:** `src/sync_dashboard.js`가 대시보드 발행(4단계) 직후 `publishDesignDocs(false)`를 **best-effort 호출**(try/catch — 문서 발행 실패가 동기화 자체를 막지 않음). 즉 "동기화" 명령 한 번으로 대시보드 + 설계서가 함께 최신화됨
- ⚠ **편집 후 발행 강제 규약:** 설계서를 수정하면 반드시 위 경로 중 하나로 GitHub를 동기화한다(문서 상단 📄 규약). 로컬만 고치고 발행을 빠뜨리면 공개본이 과거 상태로 남는다

  - **최신 연도(26년) 강조**: 보라 배경 보색 노랑(#FACC15)을 최신연도에 배정(recency 순), 굵기·불투명도 강조
  - **x축 = 선택한 출고일자만** 표시 (출고일자 필터 1~7 → x축도 1~7)
- 필터 연동:
  - **출고일자(1~31) 필터 신규** (FILTER_DEFS·filterValueLists·rowPasses·initDefaultPeriod·활성칩 추가). 출고 행만 적용, 등록과목수(과목수=출고일 null) 행은 통과 → 등록과목수는 월별 그대로
  - **월 필터 연동**: 일자별 차트가 선택 월로 필터링, 첫 화면은 기본 선택월(연도별 선 유지). 연도는 비교 위해 기본 무시
  - **편성년 필터 연동**: 사용자가 편성년을 직접 만졌을 때만 적용(`state.yearTouched`), 첫 화면은 전체 연도 비교
- 계산 로직 정합화:
  - **출고일자 필터 적용 시 등록과목수 전월/전년 증감 미산출**(값 자체는 월별 그대로). KPI 등록과목 카드 한정
  - **다중연도 전년 집합**: 선택 연도범위(span)만큼 시프트 → 25·26 선택 시 전년이 **겹치지 않는 23·24**. 단일연도(span=1)는 −12개월 유지 (`computeContext` prevYearYm)
  - **월별 추이 전월/전년 = 출고율(출고수/등록과목수) 기준**으로 변경 + "출고율 기준" 라벨 (기존 등록과목수 기준에서 전환, KPI 평균출고율과 동일 산식)
  - **센터 카드 헤더 전월/전년 = 출고율 증감으로 통일** (기존 과목증감률 → 출고율, 헤더와 그 아래 월별추이 카드 불일치 해소). 이상 징후 탭의 과목감소위험/출고감소위험 칩은 각 지표 그대로 유지
- 센터별 카드:
  - **일자별 출고수 추이 차트 추가** (월별 추이 아래, 해당 센터 한정 · `getDayTrend(rows,f,센터명)`). 인스턴스별 ID + svg 데이터속성 기반 툴팁 → 다중 카드 동시 펼침 정상
  - 센터 추이 차트 명칭 **"시간 추이" → "월별 추이"**
- 성능: 필터 시트가 열린 동안 체크할 때마다 전체 render(19만 행 다중 스캔)가 돌던 것을, **시트 닫을 때 1회만** 반영(`state._sheetDirty`). 시트가 대시보드를 가리므로 체감 변화 없이 렉 제거
